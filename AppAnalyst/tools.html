Operational Tooling Guide — CVC OEI Exchange

Postman · Jira · Basecamp — Best Practices for Application Support
Postman Jira Basecamp Integration Quick Ref

Three tools. One workflow.

The CVC OEI Exchange connects 116 California community colleges through a complex web of SIS integrations, Canvas LMS syncs, and identity management APIs. When something breaks, students lose course access. This guide documents the three operational tools that keep the Exchange running — and how they work together.

Section 1

Postman

The primary diagnostic instrument for API-level troubleshooting across the Exchange's four-layer integration stack.

1.1 — What Is Postman & Why We Use It

Postman is an API testing and debugging platform. In the context of the CVC Exchange, it is how we inspect, reproduce, and diagnose failures in the integration pipeline that connects every community college in California to shared online courses.

The Four-Layer Integration Stack

Every cross-enrollment request passes through four systems:

/* The CVC Exchange Integration Stack */

Home College SIS          CVC Exchange          Teaching College          Canvas LMS
(Banner / PeopleSoft /    API Gateway &          SIS validates            Student gains
 Colleague / Ethos)       enrollment engine      enrollment               course access
       ───────────►           ───────────►           ───────────►
  Student initiates       Exchange creates       Teaching college        Roster sync puts
  cross-enrollment        enrollment record      accepts/rejects         student in course

When a college reports "enrollments aren't syncing," the failure could be at any layer. Postman lets you test each layer independently, isolate the broken component, and document the failure with reproducible evidence that you attach directly to the Jira ticket.

Why not just check logs? Logs tell you what happened. Postman lets you reproduce what happened, test hypotheses, and verify the fix. It is active investigation, not passive observation.

Status Codes — What They Mean at CVC

CodeMeaningCVC Context

201CreatedNew enrollment record created in Exchange.
204No ContentSuccessful delete/update, no body returned.
400Bad RequestMalformed enrollment payload. Check required fields (CCCID, term, CRN).

403ForbiddenValid token but wrong scope. The credential works but lacks the specific permission for this endpoint.

Variable Naming Conventions

Use double-brace syntax in your requests: {{base_url}}/api/enrollments?collegeId={{college_id}}. This means changing environments automatically switches every request between staging and production without editing URLs.

  • {{base_url}} — API root
  • {{auth_token}} — current Bearer token
  • {{college_id}} — target college code
  • {{student_cccid}} — test student identifier
  • {{enrollment_id}} — current enrollment being investigated
1.4 — Authentication & OAuth2

OAuth2 Client Credentials Flow

The Exchange uses the client_credentials grant type for service-to-service communication. There is no user login involved — the application authenticates itself with a client ID and secret.

POST {{auth_url}}
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=cvc-exchange-support
&client_secret={{client_secret}}
&scope=enrollments.read colleges.read canvas.read

Tokens typically expire in 1 to 4 hours. The response includes expires_in (seconds). Hitting a 401 mid-diagnosis usually means your token expired while you were investigating.

Pre-request Script for Auto-refresh

// Set this at the COLLECTION level so every request inherits it
const tokenExpiry = pm.environment.get("token_expiry");
const now = Date.now();

if (!tokenExpiry || now > parseInt(tokenExpiry)) {
    pm.sendRequest({
        url: pm.environment.get("auth_url"),
        method: "POST",
        header: { "Content-Type": "application/x-www-form-urlencoded" },
        body: {
            mode: "urlencoded",
            urlencoded: [
                { key: "grant_type", value: "client_credentials" },
                { key: "client_id", value: pm.environment.get("client_id") },
                { key: "client_secret", value: pm.environment.get("client_secret") },
                { key: "scope", value: "enrollments.read colleges.read canvas.read" }
            ]
        }
    }, function(err, res) {
        if (!err) {
            const json = res.json();
            pm.environment.set("auth_token", json.access_token);
            pm.environment.set("token_expiry", now + (json.expires_in * 1000) - 60000);
            // subtract 60s buffer to avoid edge-case expiry
        }
    });
}

Setting Up OAuth2 in the Authorization Tab

  1. At the collection level, go to Authorization → Type: OAuth 2.0
  2. Grant Type: Client Credentials
  3. Access Token URL: {{auth_url}}
  4. Client ID and Secret from your environment variables
  5. Scope: enrollments.read colleges.read canvas.read
  6. Click Get New Access Token → Use Token
The Expired Credential Diagnostic When you see a 401 on ANY request, your first action should be: check when the token was issued, calculate if it's expired, and refresh. This single check resolves roughly 30-40% of "API isn't working" reports.
1.5 — Daily Diagnostic Workflow

Scenario: "Enrollments aren't syncing at Sacramento City College"

This is the most common type of support request. Here is the step-by-step diagnostic using Postman:

Step 1 — Check college health

GET {{base_url}}/api/colleges/scc/health

// Look for:
//   "status": "healthy" vs "degraded" vs "offline"
//   "lastSync": timestamp — how stale is the data?
//   "sisAdapter": "banner" — which integration layer?

Step 2 — Verify your OAuth token is valid

GET {{base_url}}/api/auth/introspect
Authorization: Bearer {{auth_token}}

// If 401 → refresh token first, then continue
// Check "exp" field to see exact expiry time

Step 3 — Look up pending enrollments

GET {{base_url}}/api/enrollments?collegeId=scc&status=pending

// How many are stuck? When were they created?
// A large batch at the same timestamp = bulk sync failure
// Scattered timestamps = intermittent connectivity issue

Step 4 — Check Canvas roster sync

GET {{base_url}}/api/canvas/courses/98765/roster

// Compare this list against the enrollment records
// Students in the enrollment but NOT in the roster = sync failure

Step 5 — Compare timestamps

Cross-reference the createdAt timestamp on the enrollment record with the lastSync from the health endpoint. If the enrollment was created after the last successful sync, the data simply hasn't been pushed yet. If the enrollment is older than the last sync, the sync ran but skipped this record — that's a data-level issue.

Step 6 — Document and attach to Jira

Save the response as an example in your Postman collection. Screenshot the full request/response (including headers and timing). Attach both to the Jira ticket with a summary of findings.

Pro tip: Name your saved examples with the Jira ticket ID. Example: CVC-4521 — SCC pending enrollments (2026-04-06). This creates a searchable diagnostic history.

    pm.response.to.have.status(200);

// 2. Response time SLA

    pm.expect(json.data).to.have.property("collegeId");

// 4. No error object in success response

    const json = pm.response.json();
    pm.expect(json.errors).to.be.undefined;
});

1.9 — Common Pitfalls
Never POST, PUT, or DELETE on production without explicit approval. This cannot be overstated. A misfire against production can create phantom enrollments, drop students from courses, or corrupt identity records. If you need to test write operations, use staging exclusively.
  • Always save responses as examples — Your diagnostic work is only valuable if the team can reference it later. Save every significant response with a descriptive name and the Jira ticket ID.
  • Use collection-level auth, not per-request — Setting authentication at the collection level means every request inherits it. Per-request auth leads to inconsistency and forgotten tokens.
  • Keep environments synced with the team workspace — If someone updates the staging URL, everyone should get the change. Use Postman's team workspace, not personal copies.
  • Never hardcode tokens — Tokens go in environment variables, never pasted directly into the Authorization header. Hardcoded tokens end up in shared collections and version history.
  • Check the console — Postman's built-in console (View → Show Postman Console) shows the actual HTTP traffic, including redirects, retries, and certificate errors that the UI might hide.
Section 2

Jira

The single source of truth for every support request from 116 colleges. Ticket states drive SLA metrics, reporting, and accountability.

2.1 — What Is Jira & How CVC Uses It

Jira is the ticket-tracking system where the entire support workflow lives. Every student issue, college outage, API failure, and enhancement request flows through Jira. It is not optional tooling — it is the operational record.

Key principles:

  • If it's not in Jira, it didn't happen. Phone calls, Slack messages, and hallway conversations mean nothing unless they're documented in a ticket.
  • SLA metrics are computed from ticket state transitions. How you move tickets through the workflow directly affects the team's SLA compliance numbers that leadership reviews.
  • Tickets are read by people who weren't there. Write every ticket as if the next person to read it has zero context. Future you is one of those people.
2.2 — Ticket Lifecycle (THE Critical Workflow)
Open
↓
In Progress
↓
Waiting on Reporter
↓
Waiting on Vendor
↓
Resolved  ←  USE THIS, not Closed
↓
Closed
CRITICAL: Use "Resolved" not "Closed" to mark tickets as done. SLA metrics are calculated based on the time from Open to Resolved. If you skip to Closed, the SLA clock behaves differently and can distort reporting. "Closed" happens after the reporter confirms the resolution or after an auto-close timer. Let the workflow handle it.

When to Use Each Transition

  • Open → In Progress: When you begin active investigation. Do this immediately when you pick up a ticket.
  • In Progress → Waiting on Reporter: When you need information from the college. Include exactly what you need in your comment — do not make them guess.
  • In Progress → Waiting on Vendor: When the issue is in the SIS vendor's hands (Ellucian for Banner, Oracle for PeopleSoft). Log the vendor ticket/case number.
  • Any → Resolved: When the issue is fixed and verified. Include a resolution summary that a new analyst could follow to understand what was wrong and what fixed it.
  • Resolved → Closed: Reporter confirms fix OR auto-close after defined period. You typically do not close tickets manually.

How Transitions Affect SLA

The SLA clock pauses when the ticket is in "Waiting on Reporter" or "Waiting on Vendor." This means: transition accurately. If you're waiting for a college to respond, move the ticket to "Waiting on Reporter" immediately so your SLA isn't penalized for their response time.

2.4 — JQL (Jira Query Language) — Power User Commands

Essential Queries

// Your open tickets
assignee = currentUser() AND status != Closed

// P1 tickets created this week
priority = Critical AND created >= startOfWeek()

// Tickets for a specific college
text ~ "Sacramento City" AND status != Closed

// Aging tickets (no updates in 3+ days)
status = "In Progress" AND updated < -3d

// Tickets waiting on reporter for 5+ days (may need follow-up or auto-close)
status = "Waiting on Reporter" AND updated < -5d

// SLA at risk: high-priority tickets in progress for 4+ hours
priority in (Critical, High) AND status = "In Progress" AND created < -4h

// Banner/Ethos issues in the last 30 days (trend analysis)
labels in (banner, ethos) AND created >= -30d ORDER BY created DESC

// Unassigned tickets (grab these during slow periods)
assignee is EMPTY AND status = Open ORDER BY priority DESC, created ASC

// Your resolved tickets this month (for standup/reporting)
assignee = currentUser() AND status = Resolved AND resolved >= startOfMonth()

// Cross-query: identity/merge issues that keep recurring
component = Identity AND created >= -90d ORDER BY created DESC
Save these as filters. Each query above should be a saved filter in your Jira account. You can then use these saved filters in dashboards, subscriptions (email alerts), and quick links.

2.6 — Automation Rules

Jira automation reduces manual overhead and prevents tickets from falling through cracks. Key rules to set up or be aware of:

  • Auto-assign based on component: If component = Canvas, assign to the Canvas specialist. If component = SSO/Auth, assign to the auth lead.
  • Auto-notify on P1 creation: When a ticket is created with priority = Critical, immediately notify the team lead and post to the Basecamp emergency project.
  • Auto-close after no response: If a ticket is in "Waiting on Reporter" for 7+ days with no update, add a comment warning the reporter, then auto-resolve after 10 days. This prevents ticket graveyards.
  • SLA escalation: If a P2 ticket has been "In Progress" for 4+ hours without an update, notify the team lead. For P1, this threshold drops to 1 hour.
  • Reminder for aging tickets: If any ticket is "In Progress" with no comment in 3 days, send the assignee a reminder notification.
2.7 — Linking Jira + Postman

The diagnostic evidence from Postman should flow directly into Jira tickets:

  • Attach response screenshots: In Postman, use the built-in screenshot button or export the response body. Attach as an image or JSON file to the Jira ticket.
  • Reference collection/request names: In your Jira comment, write: "Diagnosed via Postman: Enrollment Lifecycle → Check Pending Enrollments. See attached response CVC-4521-diag-01.json"
  • Name Postman examples with Jira IDs: This creates bidirectional traceability. From the ticket you can find the Postman evidence; from Postman you can find the ticket.
  • Include response timing: "The enrollment lookup endpoint responded in 4,200ms (SLA target: 2,000ms). This confirms the latency issue reported by the college."
2.8 — Best Practices
  • Update tickets before you leave for the day. Even a one-line note: "Waiting for Banner adapter team to investigate timeout. Will follow up tomorrow AM." Your future self and any on-call colleague will thank you.
  • Always log time spent — even rough estimates. This data drives staffing decisions and helps justify headcount when ticket volume exceeds capacity.
  • Tag every ticket with the SIS platform. Without labels, trend analysis is impossible and you cannot answer "are Banner colleges having more issues than PeopleSoft colleges?"
  • Write resolution notes for a NEW analyst. Imagine someone who started yesterday reading your resolution. Would they understand what happened and why the fix works?
  • Use @mentions sparingly but effectively. @mention someone when they need to take action or are directly relevant. Do not @mention the entire team on a P4.
  • Create saved filters for your common queries. You should never be typing the same JQL twice. Save it, name it clearly, and pin it to your dashboard.

Section 3

Basecamp

The project management and async communication layer where strategy is discussed, decisions are documented, and project-level coordination happens.

3.1 — What Is Basecamp & How CVC Uses It

Basecamp is the team's project management and communication platform. It replaced email chains for project-level discussions, providing structured, searchable, and persistent communication that does not get lost in inboxes or chat scroll.

At CVC, Basecamp serves three distinct purposes:

  • Project coordination — Tracking multi-week initiatives like "CCCID Merge Fix" or "CRM Overhaul" with to-do lists, milestones, and shared files.
  • Async team communication — Discussions that need to be permanent and searchable. Unlike Slack messages that scroll away, Basecamp message board posts are easy to find months later.
  • Decision documentation — When a team makes a decision (e.g., "we will handle CCCID duplicates by always preferring the newer record"), it lives on the Basecamp message board, not in someone's memory.
3.2 — Core Concepts

Projects

Each major initiative has its own Basecamp project. Examples: "CCCID Merge Fix," "Emergency Protocol," "CRM Overhaul," "Onboarding." Each project contains all of the tools below.

Message Board

The async discussion hub. Think of it like Slack threads, but designed for permanence. Posts are organized by topic, can be categorized, and are fully searchable. Use this for discussions that need a lasting record.

To-dos

Task lists within each project. Assign to individuals, set due dates, add notes. Unlike Jira tickets (which track external support requests), Basecamp to-dos track internal project work.

Campfire

Real-time chat within a project. Use for quick, informal questions that don't need to be permanent. If a Campfire conversation produces a decision, post the decision summary on the Message Board.

Schedule

Project milestones and deadlines. Maintenance windows, release dates, and review checkpoints live here.

Docs & Files

Shared documents, SOPs, runbooks, and reference materials. This is where the team's operational knowledge base lives outside of Confluence/Jira.

Automatic Check-ins

Recurring questions posed to the team on a schedule. Examples: "What did you work on today?" (daily), "What's your plan for this week?" (Monday). Responses create a lightweight standup record without requiring meetings.

3.3 — Communication Best Practices
  • Write for skimmers. Bold the key information. Use bullet points. Put the most important thing in the first sentence, not the last. Most people will scan, not read.
  • One topic per message. If you need to discuss the CCCID merge strategy AND the upcoming maintenance window, write two separate posts. Combined posts get confused responses.
  • @mention only people who need to act. "FYI" mentions create noise. If someone doesn't need to do anything, don't @mention them — they'll see the post in the project feed.
  • Message Board for decisions, Campfire for questions. "Should we change the SLA for P3 tickets?" goes on the Message Board. "Hey, does anyone remember the staging URL?" goes in Campfire.
  • Pin important messages. If a post contains a decision, a critical reference, or a process change, pin it so newcomers find it immediately.
  • Set notification preferences. Basecamp can generate a lot of noise. Configure it so you get immediate notifications only for @mentions and posts in projects you're actively working on. Digest mode for everything else.
3.4 — How Basecamp + Jira + Postman Work Together

The three tools occupy distinct roles. Understanding when to use each one prevents confusion and duplication:

  • Basecamp = WHERE you discuss strategy, coordinate projects, and document decisions
  • Jira = WHERE individual support tickets and technical issues are tracked
  • Postman = HOW you diagnose technical problems at the API level

Example Workflow: Pattern Detection

  1. You notice via your Jira dashboard that 5 tickets this week involve Banner 401 errors across different colleges
  2. You create a Basecamp message board post: "Pattern: Banner OAuth failures across 5 colleges this week — systemic issue?"
  3. Team discusses in the thread; decides to investigate the shared OAuth infrastructure
  4. Individual Jira tickets are linked to a parent "Epic" for the systemic issue
  5. You use Postman to test the OAuth token endpoint across all Banner colleges
  6. Findings from Postman are attached to the Jira epic and summarized in the Basecamp thread
  7. Resolution is implemented, verified via Postman, individual Jira tickets resolved, Basecamp thread updated with outcome
3.5 — Setting Up Your Basecamp

First Day Checklist

  1. Read all pinned messages in every project. These contain the decisions, ground rules, and context that govern how the team operates. Do not skip this.
  2. Identify active vs archived projects. Archived projects are historical reference. Focus your attention on active projects. Note which projects have recent activity.
  3. Find the emergency communications project. Andrea's emergency protocol project is where critical incidents are coordinated. Know where it is and how to post there before you need to.
  4. Set up Automatic Check-in responses. When the daily or weekly check-in prompt arrives, respond to it. This builds visibility and trust during your onboarding period.
  5. Set notification schedule. Configure notifications for 8 AM – 5 PM Pacific only. You should not receive Basecamp pings outside work hours unless it's an @mention in the emergency project.
3.6 — Best Practices
  • Check Basecamp BEFORE Slack or email. Decisions and project updates live in Basecamp. If you only check Slack, you will miss important context. Make Basecamp your first tab of the day.
  • Search before asking. Basecamp's search is powerful. Before posting a question, search for it. There's a good chance someone asked and answered it in the last 6 months.
  • Use To-do lists for your own work tracking. Create a personal to-do list in your onboarding project. Track your own ramp-up tasks, training completions, and goals.
  • Post weekly updates in the team project. Even a brief "Here's what I worked on, here's what I learned, here's what I'm blocked on" post builds trust and visibility with your supervisor and team.
  • Bookmark the emergency project. This is the one project where speed matters more than polish. Know how to post there quickly when an incident unfolds.
