Postman · Jira · Basecamp — Best Practices for Application Support

Three tools. One workflow.

The CVC OEI Exchange connects 116 California community colleges through a complex web of SIS integrations, Canvas LMS syncs, and identity management APIs. When something breaks, students lose course access. This guide documents the three operational tools that keep the Exchange running — and how they work together.

What Tool Do I Use?

Start here. Based on what just landed on your desk, follow the path to the right tool.

I just received a…
⚠ Technical issue (API error, sync failure) ▶
POSTMAN first → diagnose the API layer → isolate root cause → update JIRA ticket

Why Postman first? You need to confirm the failure is real and identify the exact layer before you write anything in Jira. A 401 vs a 500 leads to completely different next steps.

Steps:

  1. Open Postman — run the relevant health check or enrollment lookup
  2. Capture the response: status code, error body, response time
  3. Open the Jira ticket (or create one) and attach your Postman evidence
  4. Set priority based on scope (single student = P3, multiple = P2, college-wide = P1)
📋 Support ticket from a college ▶
JIRA first → read ticket details → if API-related → POSTMAN → update JIRA

Why Jira first? The college coordinator has already described the problem. Read it fully before touching any tool. Understand what they are seeing, what students are affected, and what they have already tried.

Steps:

  1. Open the Jira ticket — read the full description and any attachments
  2. Move the ticket to "In Progress" immediately (SLA clock is running)
  3. If the issue is API-related, open Postman to reproduce and diagnose
  4. If it is a configuration or user-error issue, respond directly in the ticket
  5. Attach all evidence and update the ticket before moving to the next issue
💬 Team discussion / strategic question ▶
BASECAMP → Message Board → @mention relevant people

Why Basecamp? Strategic discussions need a permanent, searchable home. Slack and email lose context. Basecamp Message Board posts create a lasting record that the team can reference months later.

Steps:

  1. Navigate to the appropriate Basecamp project (e.g., "CCCID Merge Fix" or "CRM Overhaul")
  2. Create a new Message Board post with a clear, specific title
  3. @mention only people who need to weigh in or take action
  4. If the discussion produces a decision, pin the post for future reference
🚨 Emergency / multiple colleges down ▶
JIRA P1 → POSTMAN rapid diagnostic → BASECAMP emergency thread → email comms

All three tools, fast. A P1 is all-hands-on-deck. Speed matters more than polish. Get the ticket created, confirm the scope with Postman, and coordinate response via Basecamp.

Steps:

  1. Create or escalate the Jira ticket to P1 CRITICAL
  2. Run Postman health checks across affected colleges — determine the blast radius
  3. Post to the Basecamp emergency project: what is down, how many affected, what you know so far
  4. Update Jira and Basecamp every 15-30 minutes until resolved
  5. After resolution: write the post-incident summary in both Jira and Basecamp
📊 Weekly status / reporting ▶
JIRA dashboard → export metrics → BASECAMP weekly update

Jira is the data source. Basecamp is the communication layer. Pull your numbers from Jira, synthesize them into a human-readable update, and share on Basecamp.

Steps:

  1. Open your Jira dashboard — review resolved tickets, aging tickets, SLA compliance
  2. Run JQL for weekly metrics: resolved >= startOfWeek() AND assignee = currentUser()
  3. Export or screenshot the relevant charts
  4. Write a Basecamp Message Board post: "Week of [date] — Support Summary"
  5. Include: tickets resolved, patterns observed, blockers, next week's focus

Section 1

Postman

The primary diagnostic instrument for API-level troubleshooting across the Exchange's four-layer integration stack.

1.1 — What Is Postman & Why We Use It

Postman is an API testing and debugging platform. In the context of the CVC Exchange, it is how we inspect, reproduce, and diagnose failures in the integration pipeline that connects every community college in California to shared online courses.

The Four-Layer Integration Stack

Every cross-enrollment request passes through four systems:

/* The CVC Exchange Integration Stack */

Home College SIS          CVC Exchange          Teaching College          Canvas LMS
(Banner / PeopleSoft /    API Gateway &          SIS validates            Student gains
 Colleague / Ethos)       enrollment engine      enrollment               course access
       ───────────►           ───────────►           ───────────►
  Student initiates       Exchange creates       Teaching college        Roster sync puts
  cross-enrollment        enrollment record      accepts/rejects         student in course

When a college reports "enrollments aren't syncing," the failure could be at any layer. Postman lets you test each layer independently, isolate the broken component, and document the failure with reproducible evidence that you attach directly to the Jira ticket.

Why not just check logs? Logs tell you what happened. Postman lets you reproduce what happened, test hypotheses, and verify the fix. It is active investigation, not passive observation.

Status Codes — What They Mean at CVC

CodeMeaningCVC Context
200OKRequest succeeded. Enrollment data returned.
201CreatedNew enrollment record created in Exchange.
204No ContentSuccessful delete/update, no body returned.
400Bad RequestMalformed enrollment payload. Check required fields (CCCID, term, CRN).
401UnauthorizedExpired credentials — the #1 issue. Sacramento City College incident was caused by this. Always check token expiry first.
403ForbiddenValid token but wrong scope. The credential works but lacks the specific permission for this endpoint.
404Not FoundStudent or course does not exist in the Exchange. Possibly not yet synced from the home college SIS.
409ConflictDuplicate CCCID detected. The identity management pain point — two records exist for one student.
422UnprocessableData validation failed. Fields are present but values are invalid (e.g., enrollment date in the past).
429Too Many RequestsRate limiting triggered. Back off and retry with exponential delay. Common during peak registration.
500Internal Server ErrorTheir SIS threw an unhandled exception. Escalate to the college's IT contact.
502Bad GatewayProxy or middleware failure between Exchange and the SIS. Check Ethos layer if applicable.
503Service UnavailableScheduled maintenance or overload. Check the maintenance calendar in Basecamp.

Environments: Staging vs Production

Variable Naming Conventions

Use double-brace syntax in your requests: {{base_url}}/api/enrollments?collegeId={{college_id}}. This means changing environments automatically switches every request between staging and production without editing URLs.

  • {{base_url}} — API root
  • {{auth_token}} — current Bearer token
  • {{college_id}} — target college code
  • {{student_cccid}} — test student identifier
  • {{enrollment_id}} — current enrollment being investigated
1.4 — Authentication & OAuth2

OAuth2 Client Credentials Flow

The Exchange uses the client_credentials grant type for service-to-service communication. There is no user login involved — the application authenticates itself with a client ID and secret.

POST {{auth_url}}
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=cvc-exchange-support
&client_secret={{client_secret}}
&scope=enrollments.read colleges.read canvas.read

Tokens typically expire in 1 to 4 hours. The response includes expires_in (seconds). Hitting a 401 mid-diagnosis usually means your token expired while you were investigating.

Pre-request Script for Auto-refresh

// Set this at the COLLECTION level so every request inherits it
const tokenExpiry = pm.environment.get("token_expiry");
const now = Date.now();

if (!tokenExpiry || now > parseInt(tokenExpiry)) {
    pm.sendRequest({
        url: pm.environment.get("auth_url"),
        method: "POST",
        header: { "Content-Type": "application/x-www-form-urlencoded" },
        body: {
            mode: "urlencoded",
            urlencoded: [
                { key: "grant_type", value: "client_credentials" },
                { key: "client_id", value: pm.environment.get("client_id") },
                { key: "client_secret", value: pm.environment.get("client_secret") },
                { key: "scope", value: "enrollments.read colleges.read canvas.read" }
            ]
        }
    }, function(err, res) {
        if (!err) {
            const json = res.json();
            pm.environment.set("auth_token", json.access_token);
            pm.environment.set("token_expiry", now + (json.expires_in * 1000) - 60000);
            // subtract 60s buffer to avoid edge-case expiry
        }
    });
}

Setting Up OAuth2 in the Authorization Tab

  1. At the collection level, go to Authorization → Type: OAuth 2.0
  2. Grant Type: Client Credentials
  3. Access Token URL: {{auth_url}}
  4. Client ID and Secret from your environment variables
  5. Scope: enrollments.read colleges.read canvas.read
  6. Click Get New Access Token → Use Token
The Expired Credential Diagnostic When you see a 401 on ANY request, your first action should be: check when the token was issued, calculate if it's expired, and refresh. This single check resolves roughly 30-40% of "API isn't working" reports.
1.5 — Daily Diagnostic Workflow

Scenario: "Enrollments aren't syncing at Sacramento City College"

This is the most common type of support request. Here is the step-by-step diagnostic using Postman:

Step 1 — Check college health

GET {{base_url}}/api/colleges/scc/health

// Look for:
//   "status": "healthy" vs "degraded" vs "offline"
//   "lastSync": timestamp — how stale is the data?
//   "sisAdapter": "banner" — which integration layer?

Step 2 — Verify your OAuth token is valid

GET {{base_url}}/api/auth/introspect
Authorization: Bearer {{auth_token}}

// If 401 → refresh token first, then continue
// Check "exp" field to see exact expiry time

Step 3 — Look up pending enrollments

GET {{base_url}}/api/enrollments?collegeId=scc&status=pending

// How many are stuck? When were they created?
// A large batch at the same timestamp = bulk sync failure
// Scattered timestamps = intermittent connectivity issue

Step 4 — Check Canvas roster sync

GET {{base_url}}/api/canvas/courses/98765/roster

// Compare this list against the enrollment records
// Students in the enrollment but NOT in the roster = sync failure

Step 5 — Compare timestamps

Cross-reference the createdAt timestamp on the enrollment record with the lastSync from the health endpoint. If the enrollment was created after the last successful sync, the data simply hasn't been pushed yet. If the enrollment is older than the last sync, the sync ran but skipped this record — that's a data-level issue.

Step 6 — Document and attach to Jira

Save the response as an example in your Postman collection. Screenshot the full request/response (including headers and timing). Attach both to the Jira ticket with a summary of findings.

Pro tip: Name your saved examples with the Jira ticket ID. Example: CVC-4521 — SCC pending enrollments (2026-04-06). This creates a searchable diagnostic history.
1.6 — Pre-request Scripts & Tests

Auto-set Authorization Header

// Pre-request Script (collection level)
pm.request.headers.add({
    key: "Authorization",
    value: "Bearer " + pm.environment.get("auth_token")
});

Auto-log Response Time for SLA Tracking

// Tests tab
pm.test("Response time is under 2000ms", function() {
    pm.expect(pm.response.responseTime).to.be.below(2000);
});

// Log to console for trend tracking
console.log(`[${new Date().toISOString()}] ${pm.info.requestName}: ${pm.response.responseTime}ms`);

Standard Assertion Suite

// Paste this in the Tests tab of critical requests

// 1. Status code check
pm.test("Status code is 200", function() {
    pm.response.to.have.status(200);
});

// 2. Response time SLA
pm.test("Response time under 2s", function() {
    pm.expect(pm.response.responseTime).to.be.below(2000);
});

// 3. Required fields present
pm.test("Response has required enrollment fields", function() {
    const json = pm.response.json();
    pm.expect(json.data).to.have.property("enrollmentId");
    pm.expect(json.data).to.have.property("studentCccid");
    pm.expect(json.data).to.have.property("status");
    pm.expect(json.data).to.have.property("collegeId");
});

// 4. No error object in success response
pm.test("No errors in response", function() {
    const json = pm.response.json();
    pm.expect(json.errors).to.be.undefined;
});
1.7 — Collections to Build in Your First Month

1. Health Check Suite

A collection that hits the health endpoint for all 116 colleges. Organize into folders by region or SIS platform. Use the Collection Runner to execute all at once. Any non-200 response gets flagged immediately.

2. Enrollment Lifecycle

A collection that walks through the complete enrollment lifecycle:

  1. Create — POST a new cross-enrollment
  2. Verify — GET the enrollment, confirm it's pending
  3. Sync — Trigger or wait for the sync cycle
  4. Confirm — GET again, confirm active status
  5. Canvas Check — Verify student appears in the course roster

3. Identity Resolution

  • Lookup student by CCCID
  • Search for potential duplicate CCCIDs
  • Check merge history for a given identity
  • Validate CCCID format and cross-reference with CCCApply

4. Canvas Integration

  • Course roster retrieval
  • Student access verification
  • Grade passback status
  • SIS import status for a given course

5. Error Reproduction Library

Save one example for each error type you encounter (401, 404, 409, 422, 500, 502). Include the full request, response, and a note about the root cause. This becomes an invaluable reference when the same error surfaces again.

1.9 — Common Pitfalls
Never POST, PUT, or DELETE on production without explicit approval. This cannot be overstated. A misfire against production can create phantom enrollments, drop students from courses, or corrupt identity records. If you need to test write operations, use staging exclusively.
  • Always save responses as examples — Your diagnostic work is only valuable if the team can reference it later. Save every significant response with a descriptive name and the Jira ticket ID.
  • Use collection-level auth, not per-request — Setting authentication at the collection level means every request inherits it. Per-request auth leads to inconsistency and forgotten tokens.
  • Keep environments synced with the team workspace — If someone updates the staging URL, everyone should get the change. Use Postman's team workspace, not personal copies.
  • Never hardcode tokens — Tokens go in environment variables, never pasted directly into the Authorization header. Hardcoded tokens end up in shared collections and version history.
  • Check the console — Postman's built-in console (View → Show Postman Console) shows the actual HTTP traffic, including redirects, retries, and certificate errors that the UI might hide.

Common Gotchas

Timezone mismatches (UTC vs Pacific) All API timestamps are in UTC (the Z suffix). California is UTC-7 (PDT) or UTC-8 (PST). When a college coordinator says "it broke at 2 PM," they mean 2 PM Pacific = 9 PM or 10 PM UTC. If you search for records "created after 2 PM" using UTC, you will miss the records by 7-8 hours. Always convert.
  • null vs missing field: "syncedAt": null means "we know this field exists but it has no value yet" (never synced). If syncedAt is entirely absent from the response, the API version may not support that field — different problem.
  • String vs integer IDs: Some SIS platforms return college IDs as strings ("042") and others as integers (42). A strict equality check (===) will fail across types. Always normalize to string when comparing.
  • Empty arrays vs null: "enrollments": [] means "we queried and found nothing." "enrollments": null may mean "the query failed silently." Treat them differently in your diagnosis.

What Is Inside a JWT (JSON Web Token)

A JWT has three parts separated by dots: header.payload.signature. The payload contains claims that tell you exactly who the token belongs to, what it can do, and when it expires.

// Raw JWT (truncated for readability)
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJjdmMtZXhjaGFuZ2Utc3VwcG9ydCIs
Imlzcy....[signature]

// DECODED HEADER
{
  "alg": "RS256",      // Signing algorithm (RSA + SHA-256)
  "typ": "JWT",        // Token type
  "kid": "key-2026-q2" // Key ID — which signing key was used
}

// DECODED PAYLOAD (the important part)
{
  "sub": "cvc-exchange-support",       // Subject: who this token identifies
  "iss": "https://auth.cvc.edu",        // Issuer: which auth server created it
  "aud": "https://api.cvc.edu",         // Audience: which API accepts it
  "iat": 1743940800,                    // Issued At: Unix timestamp (April 6, 2026 08:00 UTC)
  "exp": 1743955200,                    // Expiry: Unix timestamp (April 6, 2026 12:00 UTC)
  "scope": "enrollments.read colleges.read canvas.read",
  "client_id": "cvc-exchange-support",
  "jti": "tok-8f2a-4c91-b3e7"            // JWT ID: unique token identifier for audit
}
sub (Subject)The service account name. If this says "staging-support" and you are hitting production, the token will be rejected.
iss (Issuer)Must match the API's expected issuer. A token from staging-auth.cvc.edu will be rejected by the production API.
exp (Expiry)Unix timestamp in seconds. Convert with: new Date(exp * 1000). If current time > exp, the token is dead.
scopeSpace-separated permissions. If you need enrollments.write but only have enrollments.read, you will get a 403.

Production-Grade Pre-Request Script

This script goes beyond the basic version in 1.4. It decodes the JWT, checks the actual exp claim, uses a 5-minute buffer, logs every action, and handles all error states:

// COLLECTION-LEVEL Pre-request Script: Robust OAuth2 Token Management
// Set this at the collection level so every request inherits it.

(function() {
    const token = pm.environment.get("auth_token");
    const BUFFER_MS = 5 * 60 * 1000; // 5-minute safety buffer

    // Step 1: Check if we have a token at all
    if (!token) {
        console.log('[AUTH] No token found. Requesting new token...');
        requestNewToken();
        return;
    }

    // Step 2: Decode the JWT to check expiry
    try {
        const payload = JSON.parse(atob(token.split('.')[1]));
        const expiresAt = payload.exp * 1000; // Convert to ms
        const now = Date.now();
        const minutesLeft = Math.round((expiresAt - now) / 60000);

        if (now >= expiresAt) {
            console.warn(`[AUTH] Token EXPIRED ${Math.abs(minutesLeft)} minutes ago. Refreshing...`);
            requestNewToken();
        } else if (now >= expiresAt - BUFFER_MS) {
            console.warn(`[AUTH] Token expires in ${minutesLeft} min (within buffer). Refreshing...`);
            requestNewToken();
        } else {
            console.log(`[AUTH] Token valid. Expires in ${minutesLeft} min. Subject: ${payload.sub}`);
        }
    } catch (e) {
        console.error('[AUTH] Failed to decode JWT. Requesting new token...', e.message);
        requestNewToken();
    }

    function requestNewToken() {
        const authUrl = pm.environment.get("auth_url");
        const clientId = pm.environment.get("client_id");
        const clientSecret = pm.environment.get("client_secret");

        if (!authUrl || !clientId || !clientSecret) {
            console.error('[AUTH] Missing auth_url, client_id, or client_secret in environment!');
            return;
        }

        pm.sendRequest({
            url: authUrl,
            method: "POST",
            header: { "Content-Type": "application/x-www-form-urlencoded" },
            body: {
                mode: "urlencoded",
                urlencoded: [
                    { key: "grant_type", value: "client_credentials" },
                    { key: "client_id", value: clientId },
                    { key: "client_secret", value: clientSecret },
                    { key: "scope", value: "enrollments.read colleges.read canvas.read" }
                ]
            }
        }, function(err, res) {
            if (err) {
                console.error('[AUTH] Token request FAILED:', err);
                return;
            }
            if (res.code !== 200) {
                console.error(`[AUTH] Token request returned ${res.code}: ${res.text()}`);
                return;
            }
            const json = res.json();
            pm.environment.set("auth_token", json.access_token);
            console.log(`[AUTH] New token acquired. Expires in ${json.expires_in}s.`);
        });
    }
})();

2.11 — SLA Management: The Numbers That Matter

SLA compliance is the single most visible metric for a support team. It is the number leadership reviews, the number that determines staffing, and the number that colleges use to evaluate whether the Exchange is meeting its obligations. Understanding SLA mechanics is not optional.

What Is an SLA?

A Service Level Agreement is a contractual commitment to respond to and resolve issues within defined timeframes, based on severity. The SLA is not a suggestion — it is a promise to 116 colleges that when something breaks, it will be addressed within a predictable window.

CVC SLA Tiers

P1
Critical — Multiple colleges down or >100 students affected
Response: 30 min  |  Resolution: 4 hours
P2
High — Single college degraded or <100 students affected
Response: 2 hours  |  Resolution: 8 hours
P3
Medium — Individual student issues, isolated errors
Response: 4 hours  |  Resolution: 24 hours (1 business day)
P4
Low — Enhancements, documentation, non-urgent cleanup
Response: 1 business day  |  Resolution: 5 business days

How Jira Tracks SLA

  • Time to First Response: Measured from ticket creation to the first comment or status change by the support team. This is your "response" SLA.
  • Time to Resolution: Measured from ticket creation to transition to "Resolved." This is your "resolution" SLA.
  • SLA clock pauses: The clock pauses when the ticket is in "Waiting on Reporter" or "Waiting on Vendor." This is why accurate state transitions matter — if you are waiting for a college to respond and forget to transition the ticket, the SLA clock keeps running against you.

SLA Clock Rules

Ticket StatusSLA ClockWhy
OpenRUNNINGNobody has acknowledged it yet. The college is waiting.
In ProgressRUNNINGYou are actively working. Clock still counts against resolution target.
Waiting on ReporterPAUSEDBall is in their court. Not fair to count this against your resolution time.
Waiting on VendorPAUSEDThird party dependency. You cannot resolve what you do not control.
ResolvedSTOPPEDSLA calculation complete. Time from Open to Resolved = your resolution time.
ClosedN/APost-resolution confirmation. SLA was already calculated at Resolved.

What to Do When SLA Is About to Breach

  1. 15 minutes before breach: If you cannot resolve it in time, escalate. Post in the team Campfire: "CVC-XXXX P2 approaching 8-hour SLA. Need help with [specific blocker]."
  2. If waiting on a vendor: Make sure the ticket is in "Waiting on Vendor" (clock paused). If it is still "In Progress," transition it NOW.
  3. If you missed the SLA: Document why in the ticket. "SLA breached due to 6-hour wait for Riverside IT credential rotation. Recommendation: establish emergency contact protocol for after-hours credential issues."
  4. Pattern of breaches: If the same type of ticket consistently breaches SLA, it is a systemic issue. Raise it in Basecamp as a process improvement discussion.

Monthly SLA Reporting

What to include in your monthly SLA report for management:

  • Overall compliance rate by priority tier
  • Trend line — are we improving or degrading month over month?
  • Breach analysis — for every breached ticket, why did it breach and what would prevent it next time?
  • Top offenders — which colleges or components generate the most SLA-breaching tickets?
  • Recommendations — "If we had auto-alerting for Banner OAuth expiry, 3 of our 5 P2 breaches this month would not have occurred."

SLA Gaming: Why "Resolved" Matters

Do not game the SLA. It is technically possible to move a ticket to "Resolved" prematurely to stop the clock, then reopen it later. This is dishonest, undermines the team's metrics, and will be caught in quality reviews. If a ticket is not truly resolved, it should not be in "Resolved" status. Period.

Negotiating SLA for Systemic Issues

Sometimes the SLA target is unrealistic for a specific class of issue. Example: "Vendor X takes 48 hours to respond, but our P2 SLA says 8 hours." The correct approach:

  1. Document the pattern: collect 5+ tickets where this vendor's response time caused SLA breach
  2. Calculate the impact: "We breached SLA 8 times this quarter due to Vendor X response times"
  3. Propose a solution: "Create a separate SLA category for vendor-dependent issues with a 48-hour resolution target, OR escalate the vendor SLA in the next contract renewal"
  4. Present to leadership with data, not opinions
2.12 — Ticket Writing Masterclass: The Art of Documentation

The single biggest differentiator between a junior and senior support analyst is ticket quality. A well-written ticket saves hours of back-and-forth, enables knowledge transfer, and builds your professional reputation. A poorly-written ticket creates confusion, delays resolution, and makes you look unprepared.

The STAR Format for Ticket Descriptions

  • Situation: What is happening? Who is affected? What is the scope?
  • Task: What needs to be done? What is the expected outcome?
  • Action: What have you already done? What diagnostic steps were taken?
  • Result: What was found? What is the current state? What is the next step?

3.1 — What Is Basecamp & How CVC Uses It

Basecamp is the team's project management and communication platform. It replaced email chains for project-level discussions, providing structured, searchable, and persistent communication that does not get lost in inboxes or chat scroll.

At CVC, Basecamp serves three distinct purposes:

  • Project coordination — Tracking multi-week initiatives like "CCCID Merge Fix" or "CRM Overhaul" with to-do lists, milestones, and shared files.
  • Async team communication — Discussions that need to be permanent and searchable. Unlike Slack messages that scroll away, Basecamp message board posts are easy to find months later.
  • Decision documentation — When a team makes a decision (e.g., "we will handle CCCID duplicates by always preferring the newer record"), it lives on the Basecamp message board, not in someone's memory.
3.2 — Core Concepts

Projects

Each major initiative has its own Basecamp project. Examples: "CCCID Merge Fix," "Emergency Protocol," "CRM Overhaul," "Onboarding." Each project contains all of the tools below.

Message Board

The async discussion hub. Think of it like Slack threads, but designed for permanence. Posts are organized by topic, can be categorized, and are fully searchable. Use this for discussions that need a lasting record.

To-dos

Task lists within each project. Assign to individuals, set due dates, add notes. Unlike Jira tickets (which track external support requests), Basecamp to-dos track internal project work.

Campfire

Real-time chat within a project. Use for quick, informal questions that don't need to be permanent. If a Campfire conversation produces a decision, post the decision summary on the Message Board.

Schedule

Project milestones and deadlines. Maintenance windows, release dates, and review checkpoints live here.

Docs & Files

Shared documents, SOPs, runbooks, and reference materials. This is where the team's operational knowledge base lives outside of Confluence/Jira.

Automatic Check-ins

Recurring questions posed to the team on a schedule. Examples: "What did you work on today?" (daily), "What's your plan for this week?" (Monday). Responses create a lightweight standup record without requiring meetings.

3.3 — Communication Best Practices
  • Write for skimmers. Bold the key information. Use bullet points. Put the most important thing in the first sentence, not the last. Most people will scan, not read.
  • One topic per message. If you need to discuss the CCCID merge strategy AND the upcoming maintenance window, write two separate posts. Combined posts get confused responses.
  • @mention only people who need to act. "FYI" mentions create noise. If someone doesn't need to do anything, don't @mention them — they'll see the post in the project feed.
  • Message Board for decisions, Campfire for questions. "Should we change the SLA for P3 tickets?" goes on the Message Board. "Hey, does anyone remember the staging URL?" goes in Campfire.
  • Pin important messages. If a post contains a decision, a critical reference, or a process change, pin it so newcomers find it immediately.
  • Set notification preferences. Basecamp can generate a lot of noise. Configure it so you get immediate notifications only for @mentions and posts in projects you're actively working on. Digest mode for everything else.

4.1 — The Four SIS Platforms: What You Need to Know

The California Community College system runs on four Student Information System (SIS) platforms. Every integration issue you debug will trace back to one of these. Recognizing which SIS a college uses from the error response is a core skill.

Banner (Ellucian)
Ellucian · ~40% of CCC colleges
  • Architecture: REST API in Banner 9+. Older installations may still use Banner 8 with SOAP/XML.
  • Key tables: SFRSTCR (enrollment), SPRIDEN (identity/names), SGBSTDN (student general), SOBCURR (curriculum)
  • Common issues: Job scheduler delays (nightly batch runs at 2 AM, but if the job hangs, nothing syncs until IT intervenes), OAuth credential expiry (the #1 Banner issue)
  • API pattern: /api/v1/students/{bannerId}/registrations
  • How to recognize: Error responses reference "Banner," "BAN," or adapter type "banner" or "banner-ethos"
Banner Ethos (Ellucian)
Ellucian Ethos Integration · Middleware layer
  • Architecture: Middleware that sits between Banner and external systems. Adds an abstraction layer with its own API, authentication, and message queue.
  • Key concept: Ethos subscriptions define which data changes are broadcast to external consumers. If a subscription is misconfigured, changes in Banner do not propagate.
  • Common issues: Ethos queue backlogs (thousands of unprocessed change notifications), expired Ethos API keys (different from Banner credentials), subscription config drift after upgrades
  • How to recognize: Adapter type "banner-ethos", errors reference "Ethos," "Change Notification," or "subscription"
  • Key debugging: Ask college IT: "Is your Ethos Integration tenant healthy? Are change notification queues backing up?"
PeopleSoft (Oracle)
Oracle · Used by larger districts
  • Architecture: Integration Broker for messaging. Component Interfaces (CI) for direct data access. XML-based data exchange.
  • Key record types: STDNT_ENRL (enrollment), PERSON (identity), CLASS_TBL (course sections)
  • Common issues: Component Interface timeouts (complex queries exceed 30-second timeout), XML parsing errors (character encoding mismatches), record delimiter mismatches between PeopleSoft versions
  • API pattern: Integration Broker publishes messages to subscribers. Not a traditional REST API.
  • How to recognize: Adapter type "peoplesoft", errors reference "Integration Broker," "CI timeout," or "PSFT"
Colleague (Ellucian)
Ellucian · Older system, still in active use
  • Architecture: CXML-based API. Some colleges still use FTP-based batch file exchange for data sync.
  • Key data: STUDENT.ACAD.CRED (enrollment), PERSON (identity), SECTIONS (course sections)
  • Common issues: Character encoding (UTF-8 vs Latin-1 causes name corruption for students with accented characters), batch file processing delays (FTP drops, file format changes), CXML parsing failures
  • API pattern: CXML requests/responses or flat-file batch exchange
  • How to recognize: Adapter type "colleague", errors reference "CXML," "Colleague," or "DMI" (Datatel Management Interface, legacy name)

Quick Recognition Guide

"sisAdapter": "colleague-ethos"  → Colleague via Ethos (newer installs)

4.2 — The CCCID Problem: Identity Management in a 116-College System

If there is one technical issue that defines the CVC Exchange support experience, it is duplicate CCCIDs. Understanding this problem deeply is arguably more important than understanding any single tool.

What Is a CCCID?

The California Community College ID is the statewide unique identifier assigned to every student in the CCC system. It is intended to be a single, persistent identity that follows a student across all 116 colleges. In theory, one student = one CCCID, forever.

The CCCApply → CCCID Creation Pipeline

Student fills out CCCApply application
↓
CCCApply checks for existing CCCID (name, DOB, SSN match)
↓
Match found? → Return existing CCCID
↓
No match? → Generate new CCCID
↓
CCCID pushed to home college SIS
↓
Cross-enrollment uses CCCID as the join key across systems

How Duplicates Happen

  • Name variation: Student applies as "Maria Rodriguez" at College A and "Maria L. Rodriguez-Garcia" at College B. The matching algorithm does not find a match, so a second CCCID is created.
  • Timing: Student applies at two colleges within hours, before the first CCCID is propagated to the central system. Both applications generate new CCCIDs.
  • SSN absence: If the student does not provide an SSN on both applications (which is their right), the matching algorithm has fewer fields to work with and may fail to match.
  • Data entry errors: Typos in date of birth, transposed digits in SSN, or different email addresses prevent the system from recognizing the same person.

What Duplicate CCCIDs Break

SystemImpactSymptom
EnrollmentRecords split across two identities. The Exchange creates the enrollment under CCCID-A, but the teaching college's SIS has the student under CCCID-B.Student enrolled but "not found" at teaching college. 404 on enrollment lookup.
Canvas LMSSSO login maps to CCCID-A, but the course enrollment is under CCCID-B. Canvas cannot match the student to the roster."You are not enrolled in this course" error in Canvas.
Financial AidConsortium agreements require a single student identity. Two CCCIDs = two "students" and the consortium math breaks.Financial aid disbursement errors, consortium agreement rejection.
Grade PassbackGrades are recorded under one CCCID but the student's transcript is under the other.Missing grades on transcript. Student must petition for grade correction.

The Merge Process

  1. Identify the primary CCCID: Usually the older one with more enrollment history. Confirm with the student or college coordinator if unclear.
  2. Document in Jira: Create a ticket (or update the existing one) with both CCCIDs, evidence of the duplicate, and the recommended merge direction.
  3. Request the merge: CCCID merges are performed by the CCC Technology Center, not by the Exchange team. Submit a merge request through the established channel (varies — check with your team lead).
  4. Verify the merge: After the merge, use Postman to confirm only one CCCID exists and all enrollments are consolidated.
  5. Verify downstream: Check that Canvas access, enrollment records, and grade passback are all functioning under the merged CCCID.

Your Role

Detect, document, escalate, track. You are the detective, not the surgeon. Find the duplicate, prove it exists with Postman evidence, create the Jira ticket with full detail, request the merge through proper channels, and track it until all downstream systems are confirmed working. Do not attempt to merge records yourself unless you have been explicitly trained and authorized.
4.3 — Canvas LMS Integration: The Last Mile

The enrollment is in the Exchange. The teaching college SIS has acknowledged it. But the student still cannot access their course. The "last mile" between the Exchange and Canvas is where a surprising number of issues live.

How Exchange Enrollment Becomes Canvas Roster

Exchange enrollment status = "active"
↓
Exchange generates SIS Import CSV file
↓
CSV pushed to Canvas via SIS Import API
↓
Canvas processes CSV (batch job, may take 5-15 min)

The SIS Import API

4.4 — The Academic Calendar: Why Timing Matters for Support

Support ticket volume is not random. It follows the academic calendar with high predictability. Understanding this cycle lets you anticipate problems, pre-scale resources, and warn colleges before known pain points hit.

The 16-Week Calendar and Ticket Volume

Weeks 1-2: Registration Opens HIGH VOLUME
Enrollment surge. API rate limits hit. Capacity issues. Duplicate CCCIDs spike as students apply to multiple colleges simultaneously. This is your highest-volume period. Expect 3-5x normal ticket volume.
Week 3: Late Registration / Add ELEVATED
Students adding courses at the last minute. Enrollment creation errors. Canvas sync delays as batch sizes increase. Cross-enrollment between colleges that have different add deadlines creates timing conflicts.
Week 4: Add/Drop Deadline HIGH VOLUME
Last-minute drops and enrollment status changes. Drop processing failures. Refund calculation errors. Financial aid recalculation triggers. Students panicking about access they are about to lose.
Week 5: Census Date CRITICAL
Enrollment records are locked after census. Any sync failure that has not been resolved by census becomes a critical data integrity issue. Colleges must report accurate enrollment numbers. If the Exchange shows 50 students but the SIS shows 45, that is a crisis.
Weeks 6-10: Mid-Semester NORMAL
Lowest ticket volume. Use this time to work on P4 improvements, update knowledge base, build monitoring, and prepare for the end-of-term surge.
Week 11: Financial Aid Disbursement ELEVATED
Consortium agreement errors spike. Financial aid verification requires accurate enrollment data across colleges. If a student has a duplicate CCCID, their consortium agreement will likely fail.
Weeks 12-14: Pre-Finals NORMAL
Moderate volume. Occasional grade passback configuration issues as instructors prepare gradebooks.
Week 15-16: Finals and Grade Submission ELEVATED
Grade passback failures. Instructors submitting grades through Canvas, which must flow back to the home college SIS. Any grade passback API timeout or configuration error means students do not receive credit.
Between Terms: Term Rollover HIGH VOLUME
Massive batch updates as SIS platforms create new terms, archive old courses, and reset enrollment records. Maintenance windows for SIS upgrades. Configuration changes that break integrations. This is the second-highest risk period after registration.

Being Proactive: Using the Calendar

  • 1 week before registration: Run full health checks across all 116 colleges. Identify any degraded adapters. Alert colleges with expiring credentials. Verify API capacity with infrastructure team.
  • 1 week before census: Run a report of all pending enrollments older than 48 hours. These need urgent attention — if they are not synced by census, students may be reported as not enrolled.
  • 1 week before grade submission: Verify grade passback is configured and functioning for all courses with cross-enrolled students. Run the grade passback test endpoint for each college.
  • During term rollover: Expect SIS maintenance windows. Proactively ask colleges for their maintenance schedules. Set up enhanced monitoring for the rollover period.

Calendar-Driven Communication

Post a Basecamp message board update before each high-volume period:

Message Board — Calendar Advisory Heads-up: [Event] begins [Date]

Expected impact: [What types of tickets we expect to see]
Preparation taken: [What we have already done to prepare]
College advisory: [What colleges should check on their end]
Monitoring plan: [Enhanced monitoring schedule for this period]

If you see unusual patterns during [event], flag them in Campfire immediately.
The analyst who says "I expected this" is worth more than the analyst who says "I didn't see this coming." The academic calendar is your crystal ball. It will not predict every issue, but it will predict the category of issues with remarkable accuracy. Use it.

Caution

What NOT to Do — Lessons from the Field

Ten anti-patterns that will cost you credibility, break metrics, or create more work. Learn from others' mistakes.

✗ Don't skip the Postman diagnostic and just email the campus IT team asking "is it broken?"
You lose credibility fast. The college IT team expects you to have already done your homework. Run the health check, check the status code, and come to them with specifics: "We are seeing a 401 on your Banner adapter — can you verify the OAuth credentials?" vs "Something seems wrong, can you check?"
✗ Don't close a Jira ticket as "Closed" — use "Resolved"
SLA metrics are computed from Open to Resolved. Skipping to Closed distorts the SLA calculation and makes it impossible to accurately report resolution times to leadership. "Closed" is a terminal state that should happen automatically after the reporter confirms the fix.
✗ Don't run POST/PUT/DELETE on Production without explicit team approval
A misfire against production can create phantom enrollments, drop students from courses, or corrupt identity records. Write operations on production require verbal or written approval from the team lead. Use staging for all testing. This is a fireable offense at most organizations.
✗ Don't write Jira tickets with just "it's broken"
Every ticket must include: the college name and code, the SIS platform, the number of students affected, steps to reproduce, and any error codes or Postman evidence. A ticket that says "enrollment issue" helps no one and will bounce back to you for more detail, wasting everyone's time.
✗ Don't use Basecamp Campfire for decisions
Campfire is for quick, informal chats. Decisions belong on the Message Board where they are searchable, permanent, and organized by topic. If a Campfire conversation produces a decision, someone must post a summary on the Message Board. Otherwise it is lost in chat scroll within a week.
✗ Don't ignore the 409 Conflict (duplicate CCCID)
A 409 means two identity records exist for one student. If you close the ticket without resolving the duplicate, it will cascade into Canvas sync failures, enrollment mismatches, and potentially financial aid issues. The duplicate must be merged. See Playbook 2 for the full process.
✗ Don't manually copy-paste OAuth tokens
Use Postman pre-request scripts to auto-refresh tokens (see section 1.4). Manually pasting tokens leads to: expired tokens in shared collections, tokens accidentally committed to version history, and inconsistent auth state between team members. Automate it once and never think about it again.
✗ Don't let tickets sit in "In Progress" for 3+ days without an update
Your SLA timer is running. Even if you are waiting for something, add a comment: "Still waiting on Riverside IT. Will follow up tomorrow AM." This tells your team lead you have not forgotten, prevents the automation reminder from firing, and gives anyone reviewing the queue context on where things stand.
✗ Don't respond to P1 emergencies on Basecamp
Basecamp is async. P1 means "right now." Create the Jira P1 ticket, use Postman for rapid diagnostics, and communicate via phone or Slack (if your team uses it). Post the Basecamp update after the fire is under control. Basecamp is for coordination and documentation, not real-time incident response.
✗ Don't forget to document the fix
If the fix is not in the knowledge base, you (or a colleague) will solve the exact same problem again in 3 months and spend another hour on it. Every resolution that took more than 30 minutes to diagnose should be documented: what was wrong, how you found it, and what fixed it. Your future self will thank you.
